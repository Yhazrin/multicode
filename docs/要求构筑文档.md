# Multicode 剩余功能开发草案（去除共享 Runtime 池）

> 面向当前 `multicode` 主干能力的下一阶段开发清单。
> 目标不是继续横向堆模块，而是补齐“行为可控、执行可复盘、调度可治理”的关键闭环。

## 0. 当前项目状态判断

从当前仓库可见，`multicode` 已经具备较完整的主干能力：

- 产品主定位已经明确为：让 coding agents 成为真实队友，核心特性包括 agents、autonomous execution、skills、runtimes、multi-workspace。
- 服务端路由已按域拆分，至少包含 auth、workspace、issue、attachment、comment、agent、skill、runtime、inbox、run 等模块入口。
- 数据模型里已经具备 `Agent`、`AgentRuntime`、`AgentTaskQueue`、`Skill`、`TaskCheckpoint`、`TaskReview`、`RuntimeUsage`、`AgentMemory` 等对象。

因此，后续开发不建议再“无序铺模块”，而建议围绕以下 3 条主线推进：

1. Prompt / Skill 装配透明化
2. Task 执行过程可回放、可调试、可追责
3. Runtime / Repo / Team 之间的调度与治理能力

---

## 1. Prompt / Skill 装配预览器

### 目标
让 Agent 最终拿到的执行上下文可预览、可解释、可调试。

### 为什么值得优先做
当前系统已经有：
- `Agent.instructions`
- `Skill.content`
- `Skill.files`
- workspace / issue / runtime 这些上下文

但如果没有“最终执行上下文”的可视化预览，随着 skill 和规则增多，agent 行为会越来越难调。

### 功能范围
#### MVP
- Agent 编辑页新增 `Prompt Preview`
- Task 详情页新增 `Execution Context Preview`
- 将最终 prompt 分段展示，并标注来源：
  - system base
  - workspace context
  - team context
  - agent instructions
  - skills
  - task context
  - runtime context
- 支持复制全文
- 支持禁用/启用某个 skill 后重新预览

#### 后续增强
- prompt versioning
- prompt diff
- prompt snapshot 绑定到 task run
- prompt test run

### 数据与后端草案
可以先不加新表，通过运行时装配返回预览结果。

新增 API：
- `POST /api/agents/{id}/prompt-preview`
- `POST /api/tasks/{id}/execution-context-preview`

返回结构建议：
```json
{
  "sections": [
    {"source": "agent.instructions", "title": "Agent Instructions", "content": "..."},
    {"source": "skill:deploy-v2", "title": "Skill: deploy-v2", "content": "..."}
  ],
  "merged_prompt": "...",
  "metadata": {
    "skills_used": ["skill_1", "skill_2"],
    "workspace_id": "...",
    "runtime_id": "..."
  }
}
```

### 前端页面草案
- Agent 编辑页右侧抽屉：Prompt Preview
- Task 详情页 Tab：Execution Context

### 实现难度
中等

### 推荐优先级
P0

## 2. Task 回放与执行报告

### 目标
让每次 agent 执行都能形成一份清晰的可回放报告，便于调试、复盘和团队协作。

### 为什么值得优先做
当前系统已经有：
- task queue / task run
- task messages
- checkpoints
- review
- runtime usage

也就是说，“数据已经有了”，现在欠缺的是面向人阅读的产品层组织。

### 功能范围
#### MVP
每个 task run 生成一份报告，包含：
- 基础信息：issue、agent、runtime、开始结束时间
- 输入上下文：issue 描述、评论、依赖、相关 skill
- 执行过程：progress、messages、checkpoints
- 最终输出：result / error
- review 结果
- 关联附件/产物

#### 后续增强
- 导出 markdown / json
- 自动生成 changelog
- 自动生成 PR summary
- 失败任务自动总结 root cause

### 数据与后端草案
优先复用现有表：
- `agent_task_queue`
- `task_message`
- `task_checkpoint`
- `task_review`
- `runtime_usage`

新增 API：
- `GET /api/tasks/{id}/report`
- `GET /api/tasks/{id}/timeline`
- `GET /api/tasks/{id}/artifacts`

### 前端页面草案
Task Run Detail 页面：
- Summary
- Timeline
- Tool / Message Log
- Checkpoints
- Review
- Final Output

### 实现难度
中等

### 推荐优先级
P0

## 3. Runtime 调度规则

### 目标
让任务调度从“能跑就行”升级为“按规则选择更合适的 runtime”。

### 为什么值得做
虽然共享 runtime 池已经完成，但如果没有调度规则，后续：
- runtime 越多越混乱
- repo 访问边界容易出错
- agent 与 runtime 的关系难维护

### 功能范围
#### MVP
- Agent 可绑定 1~N 个 runtime
- Runtime 支持标签：
  - `claude`
  - `codex`
  - `mac`
  - `linux`
  - `repo-access`
  - `fast`
- 调度优先级：
  1. online
  2. not stale
  3. 标签匹配
  4. running tasks 少

#### 后续增强
- preferred / fallback runtime
- team-level runtime pool
- drain mode
- recent failure score
- provider/model policy

### 数据草案
新增表：
- `runtime_assignment_policy`

字段建议：
- `id`
- `workspace_id`
- `agent_id`
- `team_id`（可空）
- `required_tags` jsonb
- `forbidden_tags` jsonb
- `preferred_runtime_ids` jsonb
- `fallback_runtime_ids` jsonb
- `max_queue_depth`
- `created_at`
- `updated_at`

### API 草案
- `GET /api/runtime-policies`
- `POST /api/runtime-policies`
- `PATCH /api/runtime-policies/{id}`
- `DELETE /api/runtime-policies/{id}`

### 页面草案
- Agent 编辑页新增 Runtime Policy 区块
- Team 页新增 Default Runtime Pool

### 实现难度
中等

### 推荐优先级
P0

## 4. Repo Scope 与仓库绑定

### 目标
明确 task 到底操作哪个 repo、哪个 branch、哪些路径。

### 为什么值得做
一旦 runtime 不再只有你自己的一台机器，repo scope 就会变成“是否能安全调度”的前置条件。

### 功能范围
#### MVP
- Workspace 绑定多个 repo
- Issue 指定 repo / branch
- Runtime 上报本地 repo availability
- 调度时仅派给有该 repo 访问条件的 runtime

#### 后续增强
- shallow clone
- read-only / write-enabled 区分
- allowed_paths
- repo credential health check

### 数据草案
你已有 `workspace.repos`，可以继续增强为：
- `repo_id`
- `repo_name`
- `default_branch`
- `access_mode`
- `allowed_paths`
- `visibility`

新增（可选）：
- `runtime_repo_scope`

### API 草案
- `GET /api/workspaces/{id}/repos`
- `PATCH /api/workspaces/{id}/repos`
- `POST /api/issues/{id}/bind-repo`
- `GET /api/runtimes/{id}/repos`

### 页面草案
- Workspace Settings → Repos
- Issue 创建/编辑页增加 repo / branch 选择

### 实现难度
中等

### 推荐优先级
P1

## 5. Skill 版本管理

### 目标
避免 skill 被直接覆盖，导致 agent 行为漂移且无法复盘。

### 为什么值得做
当 skill 变成团队共享资产后，没有版本机制就很难定位：
- 这次 agent 为什么和上次行为不同
- 哪次 skill 改动导致执行质量下降

### 功能范围
#### MVP
- skill 更新时自动生成新版本
- agent 可选择：
  - follow latest
  - pin 某个 skill version
- task run 记录当时用到的 skill version snapshot

#### 后续增强
- version diff
- 回滚到旧版本
- 发布说明
- 按 team 绑定 skill version

### 数据草案
新增表：
- `skill_version`

字段建议：
- `id`
- `skill_id`
- `version`
- `content`
- `files_snapshot`
- `created_by`
- `change_note`
- `created_at`

### API 草案
- `GET /api/skills/{id}/versions`
- `POST /api/skills/{id}/versions`
- `POST /api/skills/{id}/pin-version`

### 页面草案
- Skill Detail → Versions
- Skill Diff Modal

### 实现难度
中等

### 推荐优先级
P1

## 7. Memory 产品化

### 目标
让 memory 从“底层能力”变成“团队经验库”。

### 为什么值得做
你已经有 `AgentMemory` 了，但还需要把它从 API 能用，升级到产品上可理解、可管理。

### 功能范围
#### MVP
memory 分三类：
- short-term
- episodic
- canonical

支持：
- 浏览 memory
- pin / archive
- 从 task 生成 memory
- 查看 memory 来源 task

#### 后续增强
- workspace memory
- team memory
- memory review
- memory decay / expiration strategy

### 数据草案
在现有 `agent_memory` 上补字段：
- `memory_type`
- `source_task_id`
- `importance`
- `is_pinned`
- `scope`

### API 草案
- `GET /api/agents/{id}/memory`
- `POST /api/agents/{id}/memory/pin`
- `POST /api/workspace/memory/promote`

### 页面草案
- Agent Memory List
- Workspace Knowledge Feed

### 实现难度
中等

### 推荐优先级
P1

## 9. 轻量自动化规则

### 目标
让系统自动触发 agent 行为，而不是全靠手动操作。

### 为什么值得做
你的系统已经有足够多的事件源：
- issue assigned
- comment mention
- task failed
- runtime offline
- review requested

所以非常适合上一个轻量规则层。

### 功能范围
#### MVP
支持以下模板规则：
- issue assigned to agent → enqueue task
- comment mention agent → enqueue task
- task failed → notify owner
- runtime offline → mark queue degraded
- issue enters review → notify reviewer

#### 后续增强
- 条件组合
- 延时执行
- dry run
- rule hit log

### 数据草案
新增表：
- `automation_rule`
- `automation_rule_run`

### API 草案
- `GET /api/automation-rules`
- `POST /api/automation-rules`
- `PATCH /api/automation-rules/{id}`
- `GET /api/automation-rules/{id}/runs`

### 页面草案
- Settings → Automation Rules

### 实现难度
中等

### 推荐优先级
P2

## 10. Snapshot / Debug 工具链

### 目标
给你和小团队一个“调 agent”所需的最小工程工具箱。

### 为什么值得做
当系统越来越复杂时，最痛苦的通常不是“功能没有”，而是“出了问题不知道为什么”。

### 功能范围
#### MVP
- 保存每次 task 的 execution snapshot
- 保存 prompt snapshot
- 保存 runtime metadata snapshot
- 支持下载 debug bundle

#### 后续增强
- compare two task runs
- reproduce task with same context
- error replay

### 数据草案
新增表：
- `task_execution_snapshot`

字段建议：
- `task_id`
- `prompt_snapshot`
- `skill_snapshot`
- `runtime_snapshot`
- `issue_snapshot`
- `created_at`

### API 草案
- `GET /api/tasks/{id}/snapshot`
- `GET /api/tasks/{id}/debug-bundle`

### 页面草案
- Task Detail → Debug

### 实现难度
中等

### 推荐优先级
P1

## 12. 不能在当前阶段优先做的内容

这些功能目前不能做：

- 公开 runtime 市场
- 计费 / 积分 / 商业结算
- 复杂企业级 RBAC
- 超重工作流编排器
- 跨 workspace 的共享资源市场

原因很简单：对个人 / 小团队场景性价比不高，还会显著拉高复杂度。
